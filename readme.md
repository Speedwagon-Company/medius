# medius

Discord escrow-style trade bot with a provider abstraction for payment detection and payouts.

Current adapter:
- Discord (`discord.py`)

Planned adapter:
- Telegram (separate adapter layer, not implemented in this iteration)

## Safety model

- Trade state is enforced by an explicit state machine.
- Payment and payout operations use idempotency keys.
- Buyer confirmation is required before release.
- Disputes lock the trade for manual review.
- Provider webhooks require signature verification and replay-safe processing.
- Direct private-key signing in bot runtime is disabled by default.

## Security/compliance boundaries

- `PAYMENT_PROVIDER=mock` is only for local development/testing.
- In `APP_ENV=production`, mock provider is blocked (fail closed).
- Custodial mode is blocked unless `CUSTODIAL_MODE_APPROVED=true`.
- Do not store private keys, seed phrases, or raw exchange credentials in DB.
- Seller payout details are encrypted at rest using `PAYOUT_ENCRYPTION_KEY`.

## Quick start

1. Create `.env` from `.env.example` and fill required values:

```bash
cp .env.example .env
```

2. Install dependencies:

```bash
pip install -r requirements.txt
```

3. Run the bot:

```bash
python src/main.py
```

## Commands

- `/starttrade` - starts escrow flow in a private channel:
  - counterparty
  - amount
  - description
  - asset/network
  - role selection (buyer/seller)
  - seller payout wallet collection
  - payment instructions + QR
  - payment check/simulate (mock provider)
  - seller delivery mark
  - buyer confirm or dispute

- `/tradeadmin` (strict admin-only):
  - `release`
  - `refund`
  - `cancel`
  - `mock_paid` (mock provider only)

## Tests and lint

Install dev tooling:

```bash
pip install -r requirements-dev.txt
```

Run tests:

```bash
PYTHONPATH=src python -m unittest discover -s tests -v
```

Run lint:

```bash
ruff check .
```
